# BlockTree 项目级快捷入口统一修复方案

## 文档目标

本文用于规划并约束 `frontend/components/block-tree.tsx` 中项目级快捷入口区域的修复工作，目标是从结构上消除“空白项目状态”和“非空项目状态”之间的行为漂移，重点覆盖以下问题：

1. 空白项目点击 `从模板添加` 无反应。
2. 空白项目先点击 `从模板添加`，再点击 `添加内容块` 后，模板弹窗延迟出现。
3. 项目级快捷入口与其对应弹窗的挂载点分裂在两个条件分支中，后续继续扩展时存在高回归风险。

本文只规划本次修复，不扩展新功能。

## 一、现状与根因

### 1.1 当前实现结构

`BlockTree` 当前在项目级快捷入口区域有两套并行渲染：

- `blocks.length === 0` 时走“空白项目”分支。
- `blocks.length > 0` 时走“正常树形视图”分支。

这两个分支里都各自渲染了项目级快捷入口按钮：

- `添加组`
- `添加内容块`
- `从模板添加`
- `从 JSON 导入`

但弹窗挂载并没有同步统一：

- `ProjectContentTreeImportModal` 已经被挂在稳定父层，因此空白/非空两种状态都能工作。
- `从模板添加内容块` 的模板选择弹窗仍只挂在非空项目分支。

### 1.2 当前真实 bug

#### A. 空白项目点击 `从模板添加` 没反应

原因不是按钮没绑事件，而是：

- 按钮会正确调用 `openTemplateModal()`
- `showTemplateModal` 也会被设为 `true`
- 但空白项目分支根本没有渲染模板弹窗

因此用户看起来像“没有任何反应”。

#### B. 点击 `添加内容块` 后，之前的模板弹窗反而出现

这是上一个 bug 的连锁结果：

- 用户先在空白项目状态把 `showTemplateModal` 改成了 `true`
- 组件仍留在空白分支，所以弹窗看不见
- 当 `添加内容块` 成功后，`blocks.length` 从 `0` 变成 `1`
- `BlockTree` 重新渲染到非空分支
- 非空分支里刚好挂着模板弹窗，所以之前残留的 `showTemplateModal=true` 被“补显示”了出来

这不是偶发问题，而是确定性的状态与挂载点错位。

### 1.3 结构性风险

当前实现有两个高风险点：

1. 项目级快捷入口按钮被写了两份，空白项目和非空项目各一份。
2. 对应弹窗没有全部挂在稳定父层，而是部分挂在某个条件分支里。

这意味着：

- 修一个分支很容易漏另一个分支
- 当前已新增的 `从 JSON 导入`、`从模板添加`、未来更多快捷入口都可能再次出现“按钮在一个分支、弹窗在另一个分支”的回归

## 二、修复原则

本次修复必须遵守以下原则：

1. 不通过补一个 `if` 或在空白分支里复制同一份模板弹窗来止血。
2. 不再维持两套项目级快捷入口 JSX。
3. 所有项目级快捷入口相关状态与弹窗挂载，都必须落到 `BlockTree` 的稳定父层。
4. 空白项目和非空项目必须共用同一套项目级快捷入口定义。
5. 保持现有业务接口不变：
   - `blockAPI.applyTemplate(...)`
   - `project-level JSON import modal`
6. 保持现有视觉与交互风格一致，不引入新的 UI 模式。

## 三、推荐修复方案

### 3.1 统一项目级快捷入口渲染

在 `BlockTree` 中抽出一套单一的项目级快捷入口渲染逻辑，例如：

- 内部辅助渲染函数
- 或轻量的本地子组件

它只负责渲染这四个入口：

1. `添加组`
2. `添加内容块`
3. `从模板添加`
4. `从 JSON 导入`

然后：

- 空白项目分支复用它
- 非空项目分支也复用它

这样以后新增项目级快捷入口时，只改一处。

### 3.2 把项目级模板选择弹窗挂到稳定父层

当前 `showTemplateModal`、`templates`、`templatesLoading`、`isApplyingTemplate`、`openTemplateModal()`、`handleAddFromTemplateTopLevel()` 都属于项目级行为，不应该和“是否已有内容块”绑定。

因此应把项目级模板选择弹窗改成：

- 始终挂在 `BlockTree` 稳定父层
- 与 `ProjectContentTreeImportModal` 同级
- 不再依赖 `blocks.length === 0` / `> 0` 的条件分支

这样可以确保：

- 空白项目点击 `从模板添加` 时立即看到弹窗
- 非空项目点击 `从模板添加` 时行为不变
- 状态不会因为切换渲染分支而延迟显现

### 3.3 保持节点级逻辑不动

本次修复只针对项目级快捷入口区域：

- `BlockNode` 内的节点级 `从模板添加` 保持现有行为
- 不改节点级菜单逻辑

这样能把改动范围控制在真实问题所在区域。

## 四、实施步骤

1. 新增文档并锁定方案。
2. 重构 `BlockTree`：
   - 抽出统一的项目级快捷入口渲染
   - 把项目级模板选择弹窗提升到稳定父层
   - 保留现有导入弹窗在稳定父层
3. 添加回归测试。
4. 跑针对性 lint / test。
5. 人工复核空白项目与非空项目两条路径的状态切换。

## 五、测试与验收

### 5.1 空白项目路径

- 空白项目点击 `从模板添加`，模板弹窗立即出现。
- 空白项目点击 `从 JSON 导入`，导入弹窗立即出现。
- 空白项目点击 `添加内容块`，不应意外触发模板弹窗。

### 5.2 非空项目路径

- 非空项目点击 `从模板添加`，模板弹窗正常出现。
- 非空项目点击 `从 JSON 导入`，导入弹窗正常出现。
- 非空项目点击 `添加组 / 添加内容块` 不影响模板弹窗状态。

### 5.3 回归约束

- 项目级快捷入口按钮只保留一套定义。
- 项目级模板选择弹窗只保留一个稳定挂载点。
- 不允许再次出现“按钮存在，但对应弹窗只在另一分支里挂载”的结构。

## 六、推荐结论

最稳妥、最符合现有项目结构的修法是：

- 不补丁式地给空白项目分支单独塞一个模板弹窗
- 而是把项目级快捷入口和项目级弹窗都统一收敛到 `BlockTree` 的稳定父层

这样能一次性解决当前 bug，并把同类回归风险一起消掉。
