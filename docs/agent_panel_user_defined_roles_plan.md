# 项目页右侧 Agent Panel 用户自定义角色改造规划

创建时间：2026-03-07  
状态：已完成（2026-03-07 已实现并完成阶段验证）  
目标：将项目页右侧 `AgentPanel` 从“预置几个固定角色供切换”改为“用户可填写、可维护、可持久化的角色信息”，并保持与现有架构统一。

---

## 1. 先说结论

这次改造的本质，不是把顶部几个按钮改成可编辑文案，也不是在现有预置模式旁边补一个“自定义角色”入口。

**根本问题是：右侧 Agent 的“角色定义”现在仍然以系统预置为运行时事实来源，而不是以用户填写的数据为运行时事实来源。**

因此，正确的改造方向应当是：

1. **把右侧 Agent 角色从“系统预置模式”升级为“项目级、用户可维护的角色配置”。**
2. **保持单一事实来源，不再并存“预置角色是真、用户填写只是补丁”的双轨逻辑。**
3. **把运行时身份标识和用户可编辑展示信息解耦。**
4. **让前端 UI、后端 API、会话隔离、记忆归因、测试基线同时切换到新事实来源。**

用奥卡姆剃刀原则收敛后，最合理的方案不是新造第三套角色系统，而是**沿现有 `AgentMode` 这条链演进**，把它从“全局预置模式表”改造成“支持项目级用户角色配置的统一模型”。

---

## 2. 第一性原理定义问题

### 2.1 右侧 Agent 角色在系统里的本质是什么

从运行机制看，右侧角色不是 UI 标签，而是以下四件事的组合：

1. **身份提示词**：决定 Agent 以什么视角思考，来源是 `system_prompt`
2. **会话隔离键**：不同角色的对话历史独立存储
3. **记忆来源标签**：记忆系统会记录这条信息来自哪个角色
4. **用户认知入口**：用户在项目页通过它切换协作视角

只改 UI 文案，而不改这四件事的事实来源，需求就没有被根本满足。

### 2.2 用户真正要的是什么

用户说“现在是预置了几个角色，我需要整体变成用户可填写角色信息的”，本质上是在要求：

- 角色不应再是代码或初始化脚本预先决定的
- 角色信息应当变成可编辑数据
- 这些数据应直接驱动右侧 Agent 的真实行为
- 用户填写后的角色应成为项目工作流的一部分，而不是设置页里一个边缘配置

### 2.3 什么叫“根本修复”

根本修复必须满足：

- 角色的增删改查由用户数据驱动
- 运行时调用只认用户配置后的角色
- 不再依赖固定五角色作为隐式默认真相
- 不把用户填写结果偷偷映射回固定枚举
- 角色修改后，对话隔离、记忆系统、提示词构造仍然一致成立

凡是不满足这些条件的做法，都属于补丁式方案。

## 3. 现状彻查结论

### 3.1 前端现状

右侧面板 `frontend/components/agent-panel.tsx` 中，顶部标签栏直接渲染 `availableModes`：

- 通过 `modesAPI.list()` 拉取 `/api/modes/`
- 用 `mode.name` 作为切换值
- 用 `mode.display_name` 和 `mode.icon` 作为展示

这说明前端本身并没有把角色写死在 React 里，但**前端只消费角色，不提供项目页内的角色定义能力**。

### 3.2 后端现状

后端已有完整的模式模型与 CRUD：

- 模型：`backend/core/models/agent_mode.py`
- API：`backend/api/modes.py`
- 前端类型/API：`frontend/lib/api.ts`

但当前设计仍然有几个根本耦合：

1. `AgentMode.name` 是全局唯一标识
2. `api/agent.py` 运行时按 `mode_name` 查 `AgentMode`
3. `Conversation.mode` 存的是字符串模式名
4. `_build_thread_id(project_id, mode, conversation_id)` 直接把模式名拼进线程身份
5. 记忆系统 `MemoryItem.source_mode` 记录的也是字符串模式名

这意味着**角色的运行时身份和用户可见命名目前是耦合的**。

### 3.3 预置来源现状

`backend/scripts/init_db.py` 会在库为空时初始化 5 个系统模式：

- `assistant`
- `strategist`
- `critic`
- `reader`
- `creative`

这些预置模式目前不是“模板”，而是事实上的默认运行时主集合。测试也把这件事固化了。

### 3.4 编排层现状

`backend/core/orchestrator.py` 中，模式系统的本质是：

- 读取 `mode_prompt`
- 用它替换 system prompt 的身份段

这意味着角色本体其实已经很清楚：**角色 = 一段可配置的身份/行为提示词 + 若干元信息**。

所以这次改造不需要新发明角色语义，只需要把“谁来定义它”从系统预置改为用户配置。

### 3.5 测试与文档现状

`backend/tests/test_memory_mode.py` 等测试已经把“5 个预置模式存在且可用”写成基线。  
这说明如果只改前端而不改测试、会话、记忆、初始化逻辑，代码库会继续保留旧事实来源，形成新的双轨。

## 4. 核心架构判断

### 4.1 不应该做什么

以下方案都不应采用：

- 在 `AgentPanel` 顶部预置角色旁边额外加一个“自定义”按钮
- 保留固定五角色作为主入口，只允许用户改显示名
- 仅把 `system_prompt` 暴露成可编辑 textarea，但运行时仍按固定角色名分支
- 新建一套与 `AgentMode` 并存的 `AgentRole` 临时前端状态
- 借用评估里的 persona 体系去承载 Agent 角色
- 让项目页直接写死几段用户输入，再在发送消息时拼接附加 prompt

这些方案的问题都一样：**没有统一事实来源，只是在旧结构外面包一层 UI。**

### 4.2 最小且根本的方案

最小且根本的方案是：

1. **继续复用 `AgentMode` 这条主链，不再新建平行角色模型。**
2. **把 `AgentMode` 从“全局预置模式”演进为“支持项目级角色配置的统一实体”。**
3. **把运行时选择从 `mode.name` 迁移到稳定 ID。**
4. **把现有 5 个系统模式降级为“可选模板”或“迁移参考”，不再作为项目页运行时真相。**

这样做的原因很简单：

- 已有模型、API、前端类型、system prompt 注入链都在 `AgentMode`
- 已有问题不在“没有角色系统”，而在“角色系统的事实来源错了”
- 直接演进现有主链，比再造新链更符合奥卡姆剃刀原则

## 5. 推荐目标结构

### 5.1 统一后的角色语义

右侧 Agent 角色应被定义为一个**项目级角色配置实体**，至少包含：

- `id`：稳定运行时标识
- `project_id`：所属项目
- `display_name`：用户看到的角色名称
- `description`：角色简介
- `system_prompt`：角色身份与行为定义
- `icon`：展示图标
- `sort_order`：排序
- `is_template` 或等价字段：仅用于区分模板与项目实例

其中：

- **运行时只依赖稳定 ID**
- **展示层依赖 `display_name` / `icon` / `description`**
- **Agent 行为依赖 `system_prompt`**

### 5.1.1 用户填写边界

为了避免把后台实现细节暴露给用户，角色字段应明确分为三类：

**用户必填**

- `display_name`：角色名称
- `system_prompt`：角色身份与行为定义

**用户可选**

- `description`：角色简介
- `icon`：展示图标

**系统生成或系统维护**

- `id`：系统生成的稳定标识
- `project_id`：由当前项目确定
- `sort_order`：通过列表排序或拖拽维护，不要求用户手填
- `is_template` 或等价字段：系统内部使用

这部分要作为明确产品约束固定下来，尤其是：

- **`description` 必须是可选项，不能做成必填**
- **用户不应手填 `id`、`project_id`、`sort_order` 这类实现字段**

### 5.2 为什么要改成项目级

从第一性原理看，右侧 Agent 角色服务的是“当前项目如何协作”，而不是“整个系统永远有哪些固定角色”。

如果角色仍是全局共享，会出现这些不合理现象：

- 一个课程项目新增的“招生转化教练”会污染另一个品牌项目
- 不同项目的语气、任务边界、上下文要求会被迫共用一套角色库
- 用户在项目页里编辑角色，却影响全部项目

因此，**角色归属应该跟项目走，而不是跟系统设置走。**

### 5.3 为什么运行时必须改用稳定 ID

当前 `name` 同时承担了：

- 唯一标识
- 展示语义
- 会话隔离键
- 记忆来源标签

这会导致一个根本问题：**用户一旦修改角色名称，就会破坏历史归属和线程身份。**

所以这次改造必须把：

- “机器用的身份”改成稳定 ID
- “人看的角色名”保留为可编辑字段

否则只是把问题从“不能编辑”变成“编辑后容易炸历史”。

## 7. 改造原则

### 7.1 单一事实来源

项目页右侧可切换角色，只能来自“当前项目保存的角色配置”。  
不能一部分来自系统预置，一部分来自用户输入。

### 7.2 模板与实例分离

如果保留现有 5 个角色，只能把它们降级为：

- 初始化模板
- 一键导入模板
- 示例角色

**绝不能继续作为运行时默认硬依赖。**

### 7.3 展示信息与运行时身份分离

用户可编辑：

- 角色名称
- 图标
- 描述
- 角色提示词

系统稳定维护：

- 角色 ID
- 会话归属
- 记忆归因
- 历史兼容

### 7.4 保持与现有结构统一

这次改造应遵循现有工程的分层：

- 前端只渲染与编辑
- API 负责 CRUD 和校验
- 编排层只消费角色定义
- 测试基线跟着事实来源一起迁移

不要把角色拼接逻辑偷偷写回 `AgentPanel` 或 `workspace/page.tsx`。

## 9. 验收标准

满足以下条件，才算需求真正完成：

1. 项目页右侧不再以固定预置角色作为默认运行时真相
2. 用户可以在项目页为当前项目创建、编辑、删除、排序角色
3. 角色修改后立即影响 Agent 的 system prompt 行为
4. 会话隔离、历史列表、重试、记忆归因在角色改名后仍然正确
5. 代码库中不再依赖“固定五角色必须存在”这一隐式前提

---

## 10. 最终建议

这次改造建议按下面的原则执行：

- **不另起炉灶**：不要新造一套脱离 `AgentMode` 的角色系统
- **不做表面可编辑**：不要只让用户改标题或 prompt，却保留旧运行时依赖
- **不保留双轨真相**：不要同时让“系统预置角色”和“用户填写角色”都成为主入口
- **先打通身份链路，再做 UI 美化**：数据模型和运行时身份不先统一，任何 UI 都只是补丁

一句话概括：

**把右侧 Agent 角色从“系统预置模式选择器”重构为“项目级用户角色配置器”，并以稳定 ID 贯通 UI、会话、记忆和 system prompt。**
