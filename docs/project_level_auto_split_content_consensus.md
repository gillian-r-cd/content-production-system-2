# 项目层级“自动拆分内容”需求共识

## 文档目的
本文用于沉淀当前关于项目层级“自动拆分内容”能力的全部讨论共识，作为后续产品方案、交互设计、技术设计与实现排期的统一基线。

本文只记录当前共识、约束、关键判断与待决策问题，不涉及代码修改。

## 一、需求的第一性原理理解

这个需求的本质，不是做一个简单的“长文本切块工具”，而是在项目层级新增一个 `内容编排编译器`：

1. 用户输入一份较长、适合分块处理的原始内容全文。
2. 系统先把原文拆成若干个带标题的块。
3. 这些拆分结果本身就是项目里的正式内容块，不是临时对象。
4. 用户在同一个弹窗内，继续围绕这些拆分结果配置完整的后续结构，包括：
   - 每个拆分块后续要挂哪些内容块/组
   - 这些内容块之间的依赖
   - 是否补空白内容块
   - 是否从内容块模板拉取结构
   - 是否增加共享内容块或共享组
   - 是否增加聚合内容块
5. 用户点击应用后，系统一次性把整套结构追加到项目中。
6. 整个弹窗阶段只做“拆分 + 编排”，不触发内容生成。
7. 后续内容生成仍然在项目中进行。

因此，这个能力的准确定位应是：

`把长原文拆成正式源内容块，并在同一界面内完成后续结构编排，然后一次性落到项目树中。`

## 二、已确认的核心产品原则

### 2.1 拆分结果是正式内容块

- 拆分后的每一块都是项目里的正式内容块。
- 它们不是预览态候选块，也不是临时草稿块。
- 点击应用后，这些块会真实进入项目结构。

### 2.2 只拆分和编排，不生成

- 该能力的目标是拆分和结构编排。
- 不在弹窗中触发 AI 生成。
- 每个内容块在编排阶段就需要把后续生成所需的要素配置完整。
- 具体生成动作仍在项目中进行。

### 2.3 单弹窗完成全部流程

- 整个体验尽量在一个弹窗中完成。
- 不拆成多个页面或多个分散入口。
- 交互感受尽量与项目内现有的补内容块、补组、拉模板、配依赖保持一致。

### 2.4 交互先做弹窗，不先做纯对话

- 第一阶段优先做弹窗形态。
- 长期可以把这套能力封装成工具，供右侧 AI Agent 调用。
- 但当前目标是先把能力本身和弹窗交互做扎实。

## 三、拆分能力的共识

### 3.1 拆分输入

弹窗中需要至少包含以下输入项：

- 原文全文输入区：用户粘贴要拆分的完整内容。
- 拆分说明输入区：用户说明拆分目的、希望怎么拆。

### 3.2 拆分模式

第一版拆分模式做成互斥选择，不混用：

1. `拆成几份`
2. `每块多少字`
3. `拆分规则`

约束如下：

- 如果只有前两种，可优先走程序化拆分。
- 如果用户填写“拆分规则”，则可以调用大模型参与语义拆分。
- 三种模式互斥，避免用户同时填多个条件导致优先级混乱。

### 3.3 标题规则

- 每个拆分结果必须有标题。
- 默认标题由系统自动生成。
- 默认形态为 `主标题 + 编号`。
- 主标题允许用户手动编辑。
- 标题先由系统生成，再允许用户修订。

### 3.4 重叠设置

- 支持上下文重叠。
- 用户可自行设置重叠多少。

### 3.5 拆分结果的最小单位

- 最小单位固定为 `标题 + 内容`。
- 暂不引入更复杂的结构，如摘要、标签、原文位置等。

## 四、chunk 的准确含义与项目树落位

为避免歧义，本文统一约定：

- `chunk` 指拆分出来的那一段文本及其标题。
- 这个 chunk 对应一个正式的“源内容块”。

但在项目树中的最终组织方式，不是让 chunk 直接裸露在同级，而是采用二层结构：

1. 每个拆分结果对应一个 `chunk 组`
2. 该组下面包含：
   - 一个“源内容块”，即该 chunk 的标题与正文
   - 该 chunk 对应的后续编排产物

换句话说：

- 拆分出来的文本本身是内容块。
- 但这个内容块需要和后续编排方案一起，被包在一个组里。

这是当前已经确认的结构共识。

## 五、编排方案的定义

### 5.1 命名

- “处理方式”统一命名为 `编排方案`。

### 5.2 编排方案的性质

- 编排方案是弹窗内使用的临时对象。
- 第一版不把它沉淀为后台可复用模板。
- 复用能力只放在后台的内容块模板体系里。
- 也就是说：
  - 弹窗里的编排方案是一次会话中的临时规则
  - 真正可复用的结构，仍通过后台模板承载

### 5.3 编排方案支持的操作

每个编排方案应支持以下类型的结构动作：

- 直接保留 chunk 源内容块
- 在 chunk 下新增组
- 在 chunk 下新增空白内容块
- 从内容块模板拉取内容块树
- 增加包裹组
- 为新增节点配置依赖
- 支持共享结构接入
- 支持聚合结构接入

## 六、配置视图与输出视图的核心判断

这是一个已经明确的重要设计判断：

- 在弹窗内配置时，按 `编排方案` 组织
- 最终写入项目树时，按 `chunk 顺序` 组织

这意味着：

### 6.1 配置视图

- 每个编排方案都是一个独立的配置窗口/面板/区域。
- 在这个区域中，可以多选它要应用到哪些已经拆出来的 chunk。
- 用户围绕“方案”进行批量配置，而不是一个 chunk 一个 chunk 地重复劳动。

### 6.2 输出视图

- 真正写入项目后，内容应固定按 chunk 顺序排列。
- 即使不同 chunk 适配不同编排方案，也不能打乱 chunk 的整体顺序。
- 项目阅读顺序始终以原文拆分顺序为主。

## 七、共享结构与聚合结构

用户已明确提出，很多内容块可能引用同一个内容块，因此在弹窗中还需要支持独立于各 chunk 的结构。

### 7.1 共享结构

共享结构指：

- 独立于所有 chunk 存在的内容块或组
- 可以被多个 chunk 的后续流程共同依赖
- 用于避免应用后还要进项目里逐个手动补依赖

共享结构的典型作用：

- 提供公共前置上下文
- 提供所有 chunk 共用的依赖节点
- 承接需要被多处复用的公共信息

### 7.2 聚合结构

还需要支持增加 `聚合内容块`。

聚合结构指：

- 依赖多个 chunk 或多个 chunk 派生结果的内容块/组
- 用于在所有 chunk 之后做归纳、汇总、整合或总输出

### 7.3 UI 上的区域划分

- 产品上可以把“共享结构”和“聚合结构”拆成独立区域。
- 但这不是硬性要求。
- 可作为信息架构上的可选实现方式，而不是必须的唯一方案。

## 八、依赖表达的共识

### 8.1 用户感受层

- UI 显示上，依赖配置应尽量和用户在项目中配置内容块依赖时的感受完全一致。
- 不应让用户感知到一套完全不同的规则体系。

### 8.2 系统实现层

虽然用户感受应尽量一致，但系统内部必须引入一层：

`草稿态依赖标识`

原因是：

- 用户在弹窗中配置依赖时，很多未来节点还没有真正创建；
- 真正的 block id 只有在“应用”时才会生成；
- 因此不能直接依赖真实项目 block id。

### 8.3 结论

- 用户界面上继续保持“像项目里一样选依赖”。
- 但系统内部必须有一层草稿态依赖标识，用于在应用时映射为正式 block id。

## 九、chunk 的人工修订权限

当前共识是：修订能力尽量做全。

第一版目标直接覆盖以下六项：

1. 改标题
2. 改内容正文
3. 调整顺序
4. 删除某个 chunk
5. 合并两个 chunk
6. 把一个 chunk 再拆开

这意味着拆分结果不仅可看，还应可真正编辑。

## 十、项目树中的排列原则

### 10.1 chunk 之间的顺序

- 固定按 chunk 顺序排列。
- 这是明确共识。

### 10.2 chunk 内部的组织

这里的关键理解已经确认：

- `chunk` 不是指一个完整的最终子树。
- chunk 指拆分出来的那段源文本。
- 在最终项目结构里：
  - 源文本内容块会放在一个 chunk 组中
  - 用户在这个组下面继续“搭积木”
- 用户在组下面如何搭积木，应尽量保留灵活性。

也就是说，当前没有要求在产品层对“chunk 组内部所有节点的固定顺序”做强约束，而是以“组下自由搭建”为主。

## 十一、应用与校验的原则

### 11.1 应用前校验

- “应用”按钮可以要求只有在草稿校验通过时才可点击。
- 如果存在错误，必须明确打出错误信息。
- 不能静默失败，也不能让用户不知道哪里有问题。

### 11.2 关于“中途失败不允许发生”

用户预期是：

- 中途某个模板实例化失败这类情况，不应在实际应用时发生。
- 应通过代码和校验机制尽量在应用前消除这类风险。

因此产品与技术上应共同遵循：

- 能前置校验的错误尽量都在应用前暴露
- 应用阶段尽量只做确定性写入

## 十二、重复打开与重复应用的语义

这一点已经明确：

- 这个弹窗应该允许再次打开。
- 弹窗中的信息应该被保存。
- 用户之后可以回来继续查看、修改、补充。

同时：

- 每一次应用拆分到项目中以后，之前已经写入项目的内容不可撤销。
- 如果用户再次打开弹窗并再次点击应用，那么新的结果是：
  - 直接追加在项目既有目录后面
  - 不影响之前已经创建的任何内容块
  - 不做覆盖，不做回滚，不做替换

结论是：

- 这是一个 `可持续编辑、可重复追加应用` 的弹窗草稿。
- 重复插入导致的内容追加，是用户自己需要负责与判断的行为。

## 十三、与模板体系的关系

- 当前弹窗中的编排方案不做后台复用。
- 结构复用能力继续放在后台“内容块模板”里。
- 弹窗内如需复用结构，应通过“从模板拉取内容块”实现。
- 后续若要增强 AI Agent 对该能力的操作，也应优先复用同一套底层能力，而不是单独再造一套对象模型。

## 十四、单弹窗应承载的信息范围

当前讨论下，单弹窗中最终至少要能承载以下信息与操作：

1. 原文全文输入
2. 拆分说明
3. 拆分模式选择
4. 主标题配置
5. 重叠设置
6. 拆分结果预览
7. 拆分结果折叠/展开
8. chunk 标题与内容编辑
9. chunk 的删改、调序、合并、再拆
10. 编排方案配置
11. 编排方案适用的 chunk 多选
12. 新增组
13. 新增空白内容块
14. 从模板拉取结构
15. 配置依赖
16. 配置共享结构
17. 配置聚合结构
18. 校验错误展示
19. 一键应用

## 十五、“全部开始”能力的共识

除“自动拆分内容”外，项目层级还应补充一个对所有项目需求都适用的体验：

- 增加一个 `全部开始` 按钮
- 点击后，所有“能开始”的内容块都开始生成
- 这里的“能开始”指所有依赖已满足的内容块
- 这些内容块应并行开始，而不是串行一个一个开始

当前关于该能力的讨论尚未展开到足够细，但最低共识已经是：

- 这是一个项目级调度能力
- 不是简单复用现有的逐个串行触发逻辑
- 它应面向所有项目内容块，而不仅仅服务于“自动拆分内容”功能

## 十六、当前已明确的实现哲学

### 16.1 用户认知要尽量延续现有项目体验

- 新功能虽然内部需要新的草稿态结构与映射层
- 但用户在 UI 上不应被迫学习一套和项目完全不同的概念
- 尤其是：
  - 内容块
  - 组
  - 模板
  - 依赖
  - 折叠/展开
  - 多选应用

都应尽可能与现有项目中的交互感受对齐

### 16.2 真正新增的是“编排草稿层”，不是另一套项目模型

- 项目里的正式对象仍然是内容块树
- 新能力增加的是一层“应用前草稿”
- 这层草稿负责承载：
  - chunk 列表
  - 编排方案
  - 共享结构
  - 聚合结构
  - 草稿态依赖标识

### 16.3 能力优先于入口形态

- 当前优先级是先把能力做对
- 弹窗只是第一阶段入口
- 长期可继续把该能力封装给 Agent 调用

## 十七、当前仍需继续拍板的关键问题

虽然共识已经很多，但若要进入完美实现，仍有一批关键问题需要继续回答。

### 17.1 弹窗草稿的保存模型

已确认：

- 弹窗可再次打开
- 信息应被保存
- 可多次追加应用

仍需明确：

- 草稿保存粒度是什么
- 按项目保存一份，还是允许多份草稿并存
- 是否需要命名
- 是否需要版本历史

### 17.2 共享结构和聚合结构的最终落位

已确认：

- 两者都需要支持

仍需明确：

- 它们在项目树里默认位于哪里
- 是否固定放在所有 chunk 组之前/之后
- 是否需要总包裹组

### 17.3 chunk 组的默认命名规则

已确认：

- chunk 自身标题由系统生成，可手动改

仍需明确：

- 包裹该 chunk 的组叫什么
- 是否也要支持统一前缀或批量改名

### 17.4 编排方案的编辑细节

已确认：

- 方案是临时对象
- 方案按配置视图组织

仍需明确：

- 方案本身是否支持排序
- 方案是否支持复制
- 方案适用 chunk 的多选区如何表达最清晰

### 17.5 应用前预览的粒度

已确认：

- 应用前需要校验

仍需明确：

- 是否要提供完整的最终项目树预览
- 还是只给出结构摘要与错误提示

### 17.6 重复追加后的管理体验

已确认：

- 重复应用会直接追加，不影响旧结构

仍需明确：

- 是否要在 UI 上强提醒“本次会追加到项目末尾”
- 是否要提供来源标识，帮助用户区分是哪次追加生成的

### 17.7 “全部开始”的调度规则

已确认：

- 所有依赖满足的块应并行开始

仍需明确：

- 并发上限
- 失败策略
- 停止策略
- 是否自动持续拉起后续新解锁的块

## 十八、当前共识的最终摘要

项目层级“自动拆分内容”能力，当前已经形成的完整共识如下：

- 拆分结果是正式内容块。
- 整个体验的本质是“拆分 + 编排”，不是“拆分 + 生成”。
- 所有内容都尽量在一个弹窗里完成。
- 交互先落地为弹窗，未来可再封装给 Agent。
- 拆分模式第一版互斥：按份数 / 按字数 / 按规则。
- 每个拆分结果必须是“标题 + 内容”。
- 标题由系统先生成，再允许用户编辑。
- 支持上下文重叠，并允许用户设置重叠量。
- 每个拆分结果会作为一个源内容块，和后续结构一起包在一个组里。
- “处理方式”统一命名为“编排方案”。
- 编排方案是弹窗内临时规则，不做后台复用。
- 真正可复用的结构继续通过后台内容块模板承载。
- 配置视图按方案组织，最终项目输出按 chunk 顺序组织。
- 需要支持共享结构与聚合结构。
- UI 感受上依赖配置要和项目中的依赖逻辑尽量一致。
- 系统内部必须引入草稿态依赖标识。
- chunk 的人工修订能力第一版尽量一步到位，覆盖改标题、改正文、调序、删除、合并、再拆。
- 应用前可要求校验通过才能点击应用；有错误时必须明确展示错误。
- 弹窗可重复打开，信息会保存。
- 每次应用都会把新的结构追加到项目中既有内容之后，不影响之前内容。
- 项目层级还需要新增“全部开始”能力，让所有依赖已满足的内容块并行开始生成。

## 十九、相关代码的彻底盘点

本节用于明确：为了解决这个需求，当前项目中哪些代码路径必须纳入统一方案，哪些地方虽然不是主路径，但若忽略会在后续造成结构不一致、行为错乱或补丁式返工。

### 19.1 前端主路径

#### `frontend/app/workspace/page.tsx`

作用：

- 项目工作台总入口
- 负责组装左侧 `ProgressPanel`、中间 `ContentPanel`、右侧 `AgentPanel`
- 持有当前项目、当前选中块、全部块列表、刷新 key 等工作区级状态

为什么相关：

- 项目级“自动拆分内容”弹窗最终要挂在工作台主页面的项目上下文中
- 弹窗需要读取当前项目、触发内容树刷新、可能向 Agent 注入消息
- 未来若把该能力封装给 Agent，这里仍是三栏联动的总入口

#### `frontend/components/progress-panel.tsx`

作用：

- 左侧项目树面板
- 加载项目全部 `ContentBlock`
- 驱动 `BlockTree`
- 当前还会在加载后调用前端串行 `runAutoTriggerChain()`

## 二十、从根上解决的总体方案

### 20.1 设计原则

本次需求不能通过以下方式实现：

- 不能只在前端弹窗里维护临时状态，应用时循环调用 `blockAPI.create()` 拼很多块
- 不能把弹窗草稿塞进 `Project` 的兼容 JSON 字段
- 不能新增一套与 `TemplateNode` 完全平行但不兼容的树结构
- 不能继续把“全部开始”做成前端串行递归调用

根本方案应满足：

1. 项目运行态仍统一使用 `ContentBlock`
2. 新增正式、持久化的“项目级草稿层”
3. 草稿层的树结构尽量与 `TemplateNode` 同构
4. 应用阶段统一由后端编译与实例化
5. 项目级调度统一收敛到后端 service，而不是留在前端递归

### 20.2 新增正式草稿层

建议新增一层正式模型，命名建议：

- `ProjectStructureDraft`

这是一个项目级、持久化的、可重复打开编辑的正式对象，不是前端本地状态。

建议字段：

- `id`
- `project_id`
- `draft_type`
- `name`
- `status`
- `source_text`
- `split_config`
- `draft_payload`
- `validation_errors`
- `last_validated_at`
- `apply_count`
- `last_applied_at`
- `created_at`
- `updated_at`

建议语义：

- `draft_type` 第一版固定为 `auto_split`
- 数据模型允许未来扩展其他项目级需求草稿
- UI 第一版可以只暴露“每个项目一份自动拆分草稿”

为什么这是根方案：

- 用户已经明确要求弹窗可重复打开，信息会被保存
- 这天然要求一层正式持久化草稿对象
- 该草稿层又不能污染项目正式内容树

### 20.3 草稿 payload 的建议结构

建议 `draft_payload` 使用 JSON，原因是：

- 当前模板体系本身就用 `root_nodes` JSON 表达树
- 草稿结构天然是嵌套、临时、会频繁整体保存的文档
- 以正式模型 + JSON payload 承载，是统一且根本的做法，不是补丁

建议 payload 结构至少包含：

```json
{
  "chunks": [
    {
      "chunk_id": "uuid",
      "title": "主标题 01",
      "content": "拆分后的正文",
      "order_index": 0
    }
  ],
  "plans": [
    {
      "plan_id": "uuid",
      "name": "编排方案 A",
      "target_chunk_ids": ["chunk-1", "chunk-3"],
      "root_nodes": []
    }
  ],
  "shared_root_nodes": [],
  "aggregate_root_nodes": [],
  "ui_state": {}
}
```

其中：

- `chunks` 承载用户可编辑的拆分结果
- `plans` 承载按方案组织的临时编排规则
- `shared_root_nodes` 承载共享结构
- `aggregate_root_nodes` 承载聚合结构

### 20.4 草稿节点不另发明，尽量对齐 `TemplateNode`

建议：

- `plans[].root_nodes`
- `shared_root_nodes`
- `aggregate_root_nodes`

都尽量沿用 `TemplateNode` 同构结构。

原因：

- 当前项目已经有 `TemplateTreeEditor`
- 后端已经有 `normalize_template_nodes()` 和 `instantiate_template_nodes()`
- 这条演进路径已经被证明是统一方向

因此，自动拆分不是再造一套树结构，而是：

`高阶草稿（chunks + plan selections） -> 编译成 TemplateNode 兼容树 -> 实例化为 ContentBlock`

### 20.5 编译链路

建议新增一层专门的编译服务，命名建议：

- `backend/core/project_structure_compiler.py`

职责：

1. 接收 `ProjectStructureDraft`
2. 生成最终要落库的根节点树
3. 生成草稿态依赖映射
4. 做完整验证
5. 输出可直接实例化的 `root_nodes`

编译后的最终树建议形态：

- 顶层新增一个“本次应用批次组”
- 该批次组内部按固定顺序组织：
  - 共享结构组（可选）
  - chunk 组 1
  - chunk 组 2
  - ...
  - 聚合结构组（可选）

每个 chunk 组内部固定包含：

- 一个源内容块
- 该 chunk 命中的所有编排方案展开后的结构片段

这样做的好处：

- 与“重复应用时直接追加到既有目录后面”的需求完全一致
- 每次应用都有自然边界
- 不必改 `ContentBlock` 运行态模型，也能在树上清楚区分一次批量应用的结果

### 20.6 依赖系统的根方案

用户界面上，依赖感受要尽量与现有项目一致。

但系统内部不能直接保存真实 block id，而必须保存草稿态引用。

建议新增统一的草稿引用语义：

- `draft_node_id`
- `chunk_id`
- `source_node_of_chunk`
- `project_block_id`（如需支持依赖项目已有块）

推荐做法：

- 编排方案、共享结构、聚合结构在编辑态都先用草稿态 ID 引用
- 编译时统一映射为最终树中的 `template_node_id`
- 应用时再由 `instantiate_template_nodes()` 映射成真实 `ContentBlock.id`

重要约束：

- 组是容器，不应作为最终生成依赖的目标
- 真正的依赖目标应是“内容节点”，即有内容/将生成内容的块
- 共享“组”的作用是包裹共享块，而不是自己成为依赖目标

### 20.7 拆分与人工修订链路

建议新增专门的拆分服务，命名建议：

- `backend/core/project_split_service.py`

职责：

1. 根据 `split_config` 执行拆分
2. 支持三种互斥模式：
   - 按份数
   - 按字数
   - 按规则
3. 自动生成默认标题

但该服务只负责“初始拆分”。

后续用户对 chunk 的：

- 改标题
- 改正文
- 调序
- 删除
- 合并
- 再拆

都属于草稿编辑，而不是二次调用拆分服务。

### 20.8 应用阶段统一事务化

建议新增专门的应用服务，命名建议：

- `backend/core/project_structure_apply_service.py`

职责：

1. 从草稿读取数据
2. 调编译器输出最终 `root_nodes`
3. 做完整校验
4. 在一个数据库事务中实例化所有 `ContentBlock`
5. 失败则整批不写入
6. 成功后增加 `apply_count`，刷新 `last_applied_at`

应用阶段应尽量只做确定性写入，不再承担复杂推断。

### 20.9 “全部开始”的根方案

当前 `runAutoTriggerChain()` 是前端递归串行，不符合需求，也不适合作为长期基础设施。

正确方向应是新增后端项目级调度服务，命名建议：

- `backend/core/project_run_service.py`
或
- `backend/core/block_scheduler.py`

职责：

1. 统一计算项目中 ready 的块
2. 支持两种模式：
   - `auto_trigger`：只启动 `auto_generate=true` 的 ready 块
   - `start_all_ready`：忽略 `auto_generate`，启动所有 ready 块
3. 根据依赖关系分层并发执行
4. 允许设置并发上限
5. 每一轮完成后继续扫描新解锁的块

同时，应把当前生成核心逻辑从 `backend/api/blocks.py` 中进一步抽离为可复用 service，避免调度器只能“内部调用 HTTP 自己”。

### 20.10 与 Agent 的长期统一

自动拆分内容不应该直接做成 Agent 特供逻辑。

正确做法是：

1. 先把草稿模型、编译器、应用器、项目级调度器做成正式后端能力
2. 弹窗作为第一入口
3. 后续再给 Agent 暴露新的项目级工具，让它去操作同一套草稿

这样：

- UI 和 Agent 不是两套平行系统
- 不会出现“弹窗能做、Agent 做不了”或“Agent 能做、UI 没法回显”的割裂

## 二十三、实施顺序建议

建议按以下顺序实施，避免返工：

1. 先补正式草稿模型、schema 与数据库接入。
2. 再做后端拆分服务、编译器、应用器。
3. 同步抽象结构树编辑器与草稿依赖选择器。
4. 再做前端单弹窗。
5. 完成后端项目级运行器，替换前端串行 auto-trigger。
6. 最后补项目复制/导入导出/删除与 Agent 收口。

这样可以保证：

- 数据层先稳
- 编译与应用先稳
- UI 不会反向绑架底层结构
- “全部开始”不会继续叠在前端递归之上
