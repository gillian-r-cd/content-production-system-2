## P0 — 致命问题（启动即崩溃）

### UTC 导入兼容性

**文件**: `backend/core/models/base.py:12`

```python
# 当前代码 — Python 3.11+ only
from datetime import UTC, datetime
```

`datetime.UTC` 是 Python 3.11 新增的常量，在 Python 3.10 运行时会直接抛出 `ImportError`，导致所有依赖 `base.py` 的模型类（几乎是全部业务模型）均无法导入，后端无法启动。

**修复方向**（暂不改，仅记录）：

```python
# 兼容 Python 3.10
from datetime import datetime, timezone

# 原来所有 UTC 用法改为 timezone.utc
datetime.now(timezone.utc)
```

或者：在 `requirements.txt` 中将 Python 版本要求从 `>=3.10` 改为 `>=3.11`，并确保实际部署环境确实是 3.11+。

## P1 — 重要问题

### LangGraph 图编译竞态条件

**文件**: `backend/core/orchestrator.py:754`

```python
async def get_agent_graph():
    global _compiled_graph, _async_checkpointer
    if _compiled_graph is not None:
        return _compiled_graph
    # ... 初始化 ...
    _compiled_graph = _graph_builder.compile(checkpointer=_async_checkpointer)
    return _compiled_graph
```

- `if _compiled_graph is not None` 到赋值之间有异步切换点
- 并发首次请求时，多个协程可能同时通过检查，触发多次编译
- 编译代价较高，且全局变量被覆盖可能引起状态不一致

**建议**: 使用 `asyncio.Lock` 保护初始化段。

---

### 路由前缀分散管理

**文件**: `backend/main.py:367+`

各路由模块在自身文件内声明 `APIRouter(prefix=...)`, `main.py` 只调用 `include_router` 而不指定前缀。这本身不会出错，但：
- 添加新路由时需要翻阅各模块文件才能知道完整路径
- 前缀命名风格不统一时容易导致 404

**建议**: 在 `main.py` 的 `include_router` 调用旁边注释完整前缀，或统一移到 `main.py` 集中管理。

## P2 — 中等问题

### Gemini thinking_budget=0 无异常处理

**文件**: `backend/core/llm.py:114`

```python
thinking_budget = settings.google_thinking_budget
thinking_kwargs = {}
if thinking_budget > 0:
    thinking_kwargs["thinking_budget"] = thinking_budget
```

文档说 `budget=0` 是为了避免纯思考模型报错，但：
- 某些 Gemini 纯思考模型可能在不传 `thinking_budget` 时仍报错
- 整个 LLM 调用路径没有针对这种错误的 catch-and-retry

**建议**: 在 LLM 调用层包一层 try-except，捕获 Gemini 特定错误并做 fallback。

---

### Token 计数使用 GPT-4 编码

**文件**: `backend/core/orchestrator.py:561`

```python
def _count_tokens_approx(messages: list) -> int:
    import tiktoken
    enc = tiktoken.get_encoding("cl100k_base")  # GPT-4 的编码
    ...
```

- `cl100k_base` 是 GPT-4 的分词器，对中文和特殊符号的计数与 Claude/Gemini 有显著差异
- 过度裁剪 → 丢失上下文；裁剪不足 → 触发模型 context length 限制
- 影响所有 Claude/Gemini 模型的消息历史修剪逻辑

**建议**: 使用粗略字符数估算（如 `len(text) / 3`）作为通用 fallback，或为不同 provider 分别选择 tokenizer。

---

### Eval Task 状态存内存

**文件**: `backend/api/eval.py:80`

```python
_TASK_RUNTIME_STATE = {}
_TASK_RUNTIME_LOCK = threading.Lock()
```

- 服务重启后所有运行中的 Task 状态丢失，前端会看到任务"消失"
- `uvicorn --workers N`（多进程）时各进程各有一份字典，状态无法共享
- 线程锁仅保护单进程内的并发，无法跨进程

**建议**: 将运行状态写入 SQLite（已有数据库），或至少在重启时从 DB 恢复。

## P3 — 低优先级问题

### 字段模板循环依赖检测未实现

**文件**: `backend/core/models/field_template.py:158`

有 `# TODO: 实现拓扑排序检测` 注释，当前允许创建存在循环依赖的字段模板。若此类模板被应用到实际项目，字段依赖解析可能陷入无限循环。

---

### pre_questions 规范化时机不一致

`normalize_pre_questions()` 和 `normalize_pre_answers()` 在不同 API 端点中的调用位置不统一——有的在写入前调用，有的在读取后调用，有的完全没有调用。这可能导致同一份数据在不同接口返回的格式略有差异，给前端解析带来不确定性。

**建议**: 统一在数据持久化（`save` 前）调用 normalize，或使用数据库触发器。

---

## P4 — 优化建议

### build_system_prompt 缺缓存

`build_system_prompt` 函数超过 470 行，每次 Agent 节点执行都会重新执行完整的字符串拼接和条件判断。项目状态很少在单次对话中大幅变化，可以考虑以 `(project_id, phase_id, locale, ...)` 为 key 缓存 prompt，状态变化时失效。

---

### API 错误响应格式不统一

部分路由返回 `raise HTTPException(status_code=..., detail=...)`, 部分返回 `JSONResponse({"error": ...})`，两者的 JSON 结构不同（`{"detail": "..."}` vs `{"error": "..."}`）。建议统一定义 error schema 并通过 FastAPI 的 `exception_handler` 集中处理。

---

## 结论

项目整体架构设计合理，代码质量中等偏上，测试覆盖较为充分。**核心阻塞点只有一个**：Python 版本兼容性（`UTC` 导入问题），必须在启动前确认运行环境为 Python 3.11+，或修改代码兼容 3.10。其余问题均为边界场景或工程优化，不影响 Python 3.11+ 环境下的核心功能运行。

建议处理顺序：**P0 → P2（Eval 状态落库）→ P1（竞态保护）→ 其余按需处理**。
