# 左侧目录栏操作补充能力规划

## 文档目标

本文用于规划左侧目录栏中的节点三点菜单补充能力，以及项目级快捷入口补充能力，并确保它们适配当前项目的数据结构、模板体系与项目级草稿体系：

1. `复制为 Markdown 格式`
2. `下载`
3. `保存为内容块模板`
4. 项目级 `从 JSON 导入`

其中：

- `下载` 需要提供二级菜单：
  - `Markdown`
  - `JSON`
- `保存为内容块模板` 不能只支持目录节点，还必须覆盖：
  - 项目级
  - 组级
  - 内容块级
- 项目级 `从 JSON 导入` 不放在节点三点菜单里，而是和左侧底部现有项目级快捷入口放在一起：
  - `添加组`
  - `添加内容块`
  - `从模板添加`
  - `从 JSON 导入`

本文只做方案规划，不涉及代码修改。

## 一、现状基线

### 1.1 当前左侧目录栏现状

当前左侧树主要由以下组件承载：

- `frontend/components/progress-panel.tsx`
- `frontend/components/block-tree.tsx`

已确认现状：

- 项目级目前只有按钮入口，没有项目级三点菜单。
- 项目级当前已有底部快捷入口：
  - `添加组`
  - `添加内容块`
  - `从模板添加`
- 组级 / 内容块级已有三点菜单，但仅覆盖：
  - 重命名
  - 添加空白内容块
  - 从模板添加
  - 添加子组
  - 生成内容
  - 复制
  - 删除
- 当前没有：
  - 复制为 Markdown
  - 下载 Markdown / JSON
  - 保存为内容块模板
  - 项目级从 JSON 导入

### 1.2 当前项目格式的三个“真相格式”

要让新增能力适配当前项目格式，必须先明确本项目已有的三类真实结构：

#### A. 项目运行态真相：`ContentBlock`

当前项目内容树运行时使用 `ContentBlock` 表达，核心特征如下：

- 节点类型统一为 `group | field`
- 树形层级由 `parent_id` / `children` 表达
- 内容块运行态还包含：
  - `content`
  - `ai_prompt`
  - `pre_questions`
  - `pre_answers`
  - `depends_on`
  - `special_handler`
  - `need_review`
  - `auto_generate`
  - `model_override`

这决定了：

- 左侧树的项目级 / 组级 / 内容块级导出，本质上都应从 `ContentBlock` 子树出发。

#### B. 可复用结构真相：`FieldTemplate.root_nodes`

当前“内容块模板”的可复用结构真相是 `FieldTemplate.root_nodes`，它本质上也是树：

- 节点类型为 `group | field`
- 依赖用 `depends_on_template_node_ids`
- 可保留：
  - `name`
  - `ai_prompt`
  - `content`
  - `pre_questions`
  - `constraints`
  - `need_review`
  - `auto_generate`
  - `model_override`
  - `special_handler`

这决定了：

- “保存为内容块模板”不应新造第四种格式。
- 正确方向是把项目树中的选定范围，转换为 `FieldTemplate.root_nodes`。

#### C. 项目完整导出真相：`/api/projects/{id}/export`

当前项目级已有完整 JSON 导出接口，导出内容包括：

- `project`
- `content_blocks`
- `project_structure_drafts`
- 对话、版本、评分器、记忆等外围数据

这决定了：

- 项目级下载 JSON 可以直接复用现有项目导出能力。
- 但组级 / 内容块级下载 JSON 不能直接复用完整项目导出，需要新增“范围导出”格式。

## 二、目标交互

## 2.1 项目级菜单

在左侧栏项目标题区域新增项目级三点菜单，建议放在项目名右侧或项目操作按钮同一行。

菜单项：

1. `复制为 Markdown 格式`
2. `下载 >`
3. `保存为内容块模板`

`下载 >` 的二级菜单：

- `Markdown`
- `JSON`

项目级语义：

- 作用范围是当前项目的整棵 `ContentBlock` 树
- 不受节点折叠状态影响
- 默认按当前树顺序导出

## 2.2 组级菜单

在 `block-tree` 现有组节点三点菜单中补充：

1. `复制为 Markdown 格式`
2. `下载 >`
3. `保存为内容块模板`

组级语义：

- 作用范围是当前组节点及其全部后代
- 保留组作为结构根节点
- 对内部依赖、子树顺序做完整保留

## 2.3 内容块级菜单

在 `block-tree` 现有内容块节点三点菜单中补充：

1. `复制为 Markdown 格式`
2. `下载 >`
3. `保存为内容块模板`

内容块级语义：

- 作用范围是当前内容块节点
- 若未来允许内容块挂子节点，则默认按“当前节点及后代”导出；当前阶段可按单节点实现
- 保存为模板时，允许单个 `field` 直接成为模板根节点

## 三、三项能力的分层适配方案

## 3.1 复制为 Markdown 格式

### 目标

给用户一个面向阅读、分享、粘贴的轻量导出结果，不是备份格式，不承担完整恢复职责。

### 通用规则

- 基于当前树顺序输出
- 基于 `group | field` 结构输出标题层级
- 默认不导出运行态噪音信息：
  - `id`
  - `project_id`
  - `status`
  - `created_at`
  - `updated_at`
  - `deleted_at`
- 默认不导出后台外围对象：
  - 对话记录
  - 日志
  - 评分记录
  - 版本历史
- `content` 若本身就是 Markdown，则原样保留
- 空内容块保留标题，不强行伪造正文

### 项目级 Markdown 方案

项目级输出一份完整阅读稿，建议格式：

- 项目名作为一级标题
- 顶层组作为二级标题
- 组内子组继续按层级降级
- 内容块输出为对应标题 + 正文

建议序列化规则：

- 项目标题：`# 项目名`
- 顶层组：`## 组名`
- 深层组：`###/#### ...`
- 内容块：
  - 若有正文：输出标题后接正文
  - 若无正文：只输出标题

说明：

- 项目级 Markdown 的目标是“可阅读”，不是“可回导”。
- 若项目包含很多自动拆分批次组，也按当前树结构如实输出，不做额外扁平化。

### 组级 Markdown 方案

组级导出只输出该组子树：

- 当前组名作为一级标题
- 子组和内容块按内部相对深度继续展开

用途：

- 快速复制一个专题结构
- 对外发送某个目录分支
- 把一组内容贴给 LLM、文档工具或协作者

### 内容块级 Markdown 方案

内容块级导出建议最简：

- 标题：当前内容块名
- 正文：当前 `content`

建议格式：

- `# 内容块名`
- 空行
- 内容正文

若该块暂无正文：

- 仍可复制标题
- 正文为空

这与当前编辑器中“复制全文（Markdown格式）”的心智保持一致，但入口扩展到了左侧树。

## 3.2 下载

## 3.2.1 下载 Markdown

下载 Markdown 与“复制为 Markdown 格式”使用同一套序列化器，只是输出方式不同：

- 复制：写入剪贴板
- 下载：触发浏览器下载

建议文件名规则：

- 项目级：`{projectName}.md`
- 组级：`{groupName}.md`
- 内容块级：`{blockName}.md`

## 3.2.2 下载 JSON

JSON 不是给阅读用的，而是给系统间流转、排查、导入、后续自动化用的。

这里必须分级处理，不能强行三层共用一个 JSON 外壳。

### 项目级 JSON 方案

项目级 JSON 直接复用现有完整项目导出接口：

- 复用：`/api/projects/{project_id}/export`
- 保持当前完整项目导出格式

原因：

- 这是当前项目级 JSON 的既有真相格式
- 已覆盖：
  - 项目本体
  - `content_blocks`
  - `project_structure_drafts`
  - 其他项目外围数据

这意味着：

- 项目级下载 JSON 是“完整备份/迁移导出”
- 不是“仅内容树轻量导出”

### 项目级从 JSON 导入方案

你这次新增的要求，本质上不是“导入一个新项目”，而是：

- 在当前项目上下文中
- 读取一个 JSON 文件
- 将其中的目录树内容追加到当前项目末尾
- 不影响现有目录结构

因此它应被定义为：

- `项目级内容树追加导入`

而不是现有的：

- `完整项目导入并新建项目`

#### 首要兼容格式

第一优先级必须兼容当前系统自己的完整项目导出 JSON，也就是：

- `GET /api/projects/{project_id}/export` 的返回格式

原因：

- 这是当前系统里已经存在的项目 JSON 真相格式
- 用户说的“从 json 文件导入项目”，在当前语境下最自然就是导入这一类文件

#### 导入时实际消费的字段

虽然输入文件是“完整项目 JSON”，但项目级追加导入到现有项目时，不应把源项目本体整体覆盖过来。

因此第一版建议只消费以下部分：

- `content_blocks`

可选保留但默认不导入：

- `project_structure_drafts`
- `chat_messages`
- `content_versions`
- `memory_items`
- 其他外围对象

原因：

- 这次目标是“追加目录树”
- 不是“把另一个项目完整并入当前项目”
- 若连草稿、对话、评分器、记忆一并导入，会把语义从“目录合并”变成“项目合并”，复杂度和风险都会急剧上升

#### 导入后的落位规则

这是本次需求最关键的行为约束，必须固定：

- 导入结果只追加在当前项目顶层目录的末尾
- 不覆盖已有顶层节点
- 不重排已有顶层节点
- 不修改已有节点内容
- 不修改已有依赖关系

也就是说，导入前的项目树保持原样，仅在尾部新增一批根节点。

#### 导入时的 ID 与依赖处理

从项目 JSON 导入时：

- 所有导入节点都必须重新生成新的 `ContentBlock.id`
- 导入子树内部的 `parent_id` 与 `depends_on` 需要整体重映射到新 ID
- 若源 JSON 中存在指向源项目树外部的依赖：
  - 第一版建议忽略这些外部依赖
  - 并在导入结果中给出“已忽略范围外依赖”的提示

这样可以保证：

- 导入进来的这一批内容能在当前项目中自洽落地
- 不会污染当前项目既有依赖图

#### 顶层顺序规则

导入追加时，建议以“导入文件中的顶层顺序”为准，整体插入到当前项目最后一个顶层节点之后。

例如：

- 当前项目顶层有 `A / B / C`
- 导入文件顶层有 `X / Y`

则导入后变为：

- `A / B / C / X / Y`

而不是：

- 插到中间
- 替换重名节点
- 按名称自动合并

#### 重名策略

第一版建议：

- 允许重名
- 不自动合并
- 不自动重命名

原因：

- 当前项目里本来就可能存在重名内容块
- “从 JSON 导入后追加”比“智能合并”更符合可预期原则
- 自动合并最容易造成静默破坏

#### UI 入口位置

这个入口不建议放进项目级三点菜单，而应放在左侧底部现有项目级快捷入口中，与以下三个现有动作并列：

- `添加组`
- `添加内容块`
- `从模板添加`
- `从 JSON 导入`

原因：

- 这是一个“往当前项目里追加结构”的动作
- 心智上更接近“加组 / 加内容块 / 从模板加”
- 不属于“导出 / 复制 / 保存模板”这一类操作

#### 前端交互建议

- 点击 `从 JSON 导入`
- 打开文件选择器
- 读取本地 `.json` 文件
- 先做格式校验
- 校验通过后显示导入摘要：
  - 顶层节点数
  - 总节点数
  - 是否有范围外依赖
- 用户确认后执行追加导入

#### 与现有项目导入的边界

必须明确区分两条能力：

- 顶部全局入口的 `导入项目`
  - 语义：创建新项目 / 导入为独立项目
- 左侧项目级快捷入口的 `从 JSON 导入`
  - 语义：把 JSON 里的内容树追加到当前项目

两者不能混用同一产品文案，否则用户会误以为会覆盖当前项目。

### 组级 JSON 方案

组级需要新增“内容树范围导出”格式，建议命名为：

- `content_block_bundle`

建议结构：

```json
{
  "export_version": "1.0",
  "type": "content_block_bundle",
  "scope": "group",
  "source_project_id": "xxx",
  "source_root_block_id": "xxx",
  "content_blocks": [],
  "meta": {
    "root_name": "某个组",
    "node_count": 12,
    "field_count": 8,
    "group_count": 4,
    "has_external_dependencies": false
  }
}
```

其中：

- `content_blocks` 使用当前 `ContentBlock` 运行态字段集合
- 但只保留所选组子树范围内的节点
- 父子关系保留为当前运行态结构

对外部依赖的处理建议：

- 若组内节点依赖了组外节点：
  - JSON 仍允许导出
  - 但在 `meta` 中标记 `has_external_dependencies=true`
  - 并列出缺失依赖摘要
- 不自动把整个项目都拉进来，避免组级导出失控膨胀

### 内容块级 JSON 方案

内容块级也使用同一类 `content_block_bundle`，但 `scope` 为 `field`：

```json
{
  "export_version": "1.0",
  "type": "content_block_bundle",
  "scope": "field",
  "source_project_id": "xxx",
  "source_root_block_id": "xxx",
  "content_blocks": [],
  "meta": {
    "root_name": "某个内容块",
    "node_count": 1
  }
}
```

这样做的好处：

- 项目级 JSON 继续保持既有完整格式
- 组级 / 内容块级 JSON 走统一的范围导出格式
- 语义清楚，不会把“项目备份”和“子树抽取”混成一种文件

## 3.3 保存为内容块模板

## 核心原则

此功能的本质不是导出项目 JSON，而是把当前选中范围转换为：

- `FieldTemplate.root_nodes`

也就是说，“保存为内容块模板”必须走模板格式，而不是项目运行态格式。

### 统一转换规则

从 `ContentBlock` 转 `FieldTemplate.root_nodes` 时：

保留：

- `name`
- `block_type`
- `ai_prompt`
- `content`
- `pre_questions`
- `constraints`
- `depends_on_template_node_ids`
- `special_handler`
- `need_review`
- `auto_generate`
- `model_override`
- `children`

丢弃：

- `id`
- `project_id`
- `parent_id`
- `depth`
- `order_index`
- `status`
- `pre_answers`
- `created_at`
- `updated_at`
- 删除态/版本态信息

依赖转换规则：

- 若依赖目标在当前保存范围内：
  - 转换为 `depends_on_template_node_ids`
- 若依赖目标在当前保存范围外：
  - 第一版建议不写入模板依赖
  - 同时在保存弹窗中明确提示“存在范围外依赖，未写入模板”

这样可以避免生成“无法实例化的半残模板”。

### 项目级保存为内容块模板

项目级保存模板时，作用范围是整个项目内容树。

输出结果：

- 生成一个新的 `FieldTemplate`
- `root_nodes` 等于当前项目所有顶级块转换后的模板树

建议交互：

- 点击后打开“保存为内容块模板”弹窗
- 用户填写：
  - 模板名称
  - 模板描述
  - 分类
- 默认名称建议：
  - `{项目名} 模板`

项目级保存模板的意义：

- 把整个项目的结构与预置内容沉淀为可复用模板
- 特别适用于：
  - 已完成结构设计的项目
  - 已通过自动拆分编排形成稳定树结构的项目

### 组级保存为内容块模板

组级保存模板时，作用范围是当前组子树。

输出结果：

- 生成一个新的 `FieldTemplate`
- `root_nodes` 以当前组选中的根组作为模板根

建议默认名称：

- `{组名} 模板`

组级保存是最自然的模板沉淀入口，预计会是使用频率最高的一层。

### 内容块级保存为内容块模板

内容块级保存模板时，作用范围是当前内容块。

输出结果：

- 若当前节点为 `field`：
  - 生成单节点模板
- 若未来该节点下允许挂子树：
  - 则按“当前节点及后代”整体转换

建议默认名称：

- `{内容块名} 模板`

这能覆盖“把某个高质量单块结构沉淀成模板”的场景。

## 四、为什么项目级也必须允许保存为内容块模板

这是本次需求里的关键点，必须明确。

原因如下：

1. 当前模板体系的结构真相已经是树，而不是单字段模板。
2. `FieldTemplate.root_nodes` 本身就能承载多根节点、组、字段、依赖与预置内容。
3. 当前自动拆分草稿编译后落地的也是树，项目级沉淀模板在结构上天然可行。
4. 用户的真实需求不是“只保存某个节点”，而是“把已经搭好的整套结构复用出去”。

因此：

- 项目级“保存为内容块模板”不是例外需求
- 它恰恰是当前模板树体系的自然延伸

## 五、前端方案

## 5.1 菜单信息架构

### 项目级

在 `progress-panel` 新增项目级三点菜单：

- `复制为 Markdown 格式`
- `下载 >`
- `保存为内容块模板`

同时，在左侧底部现有项目级快捷入口中新增：

- `从 JSON 导入`

项目级快捷入口最终建议为：

- `添加组`
- `添加内容块`
- `从模板添加`
- `从 JSON 导入`

### 组级 / 内容块级

在 `block-tree` 的现有菜单中插入新项，建议放在：

- `复制` 之后
- `删除` 之前

建议顺序：

1. 重命名
2. 现有创建类动作
3. 生成内容
4. 复制
5. 复制为 Markdown 格式
6. 下载 >
7. 保存为内容块模板
8. 删除

## 5.2 下载二级菜单

当前 `block-tree` 菜单是手写浮层，不是现成的分层菜单组件。

因此本次规划建议：

- 统一抽象一个可支持二级菜单的轻量菜单组件
- 项目级与节点级共用

原因：

- 如果继续在 `block-tree.tsx` 内手写条件分支，菜单复杂度会快速失控
- 本次新增二级菜单只是第一波，后续很可能继续扩展

## 5.3 保存模板弹窗

“保存为内容块模板”建议统一走弹窗，而不是点一下直接保存。

弹窗字段：

- 模板名称
- 模板描述
- 模板分类
- 范围摘要
  - 项目级 / 组级 / 内容块级
  - 节点数量
  - 内容块数量
  - 是否存在范围外依赖

弹窗中的关键提示：

- 会基于当前选中范围创建新的内容块模板
- 运行态状态不会写入模板
- 范围外依赖不会自动带入模板

## 六、后端方案

## 6.1 新增三类能力

### A. Markdown 序列化服务

建议新增统一服务，例如：

- `backend/core/content_tree_export_service.py`

职责：

- 把项目树 / 子树序列化为 Markdown
- 提供：
  - 项目级序列化
  - 子树级序列化

### B. 范围 JSON 导出服务

继续放在同一服务或相邻服务中，负责：

- 项目级复用现有 `projects/export`
- 组级 / 内容块级输出 `content_block_bundle`

### C. 子树转模板服务

建议新增统一转换服务，例如：

- `backend/core/content_block_template_export_service.py`

职责：

- 从 `ContentBlock` 子树生成 `FieldTemplate.root_nodes`
- 修复内部依赖映射
- 标记范围外依赖
- 产出可直接落库的模板 payload

### D. 项目 JSON 追加导入服务

建议新增统一导入服务，例如：

- `backend/core/content_tree_import_service.py`

职责：

- 读取当前系统导出的项目 JSON
- 抽取其中的 `content_blocks`
- 重新生成 ID 并修复内部父子关系与内部依赖
- 以“追加到当前项目末尾”的方式写入当前项目
- 输出导入摘要与告警信息

## 6.2 API 规划

### 项目级

建议新增项目侧导出接口：

- `GET /api/projects/{project_id}/export-markdown`
- 继续复用：
  - `GET /api/projects/{project_id}/export`

建议新增项目级保存模板接口：

- `POST /api/projects/{project_id}/save-as-field-template`

建议新增项目级追加导入接口：

- `POST /api/projects/{project_id}/import-content-tree-json`

### 组级 / 内容块级

建议在 `blocks` 域新增范围导出接口：

- `GET /api/blocks/{block_id}/export-markdown`
- `GET /api/blocks/{block_id}/export-json`
- `POST /api/blocks/{block_id}/save-as-field-template`

这样做的原因：

- 组和内容块在运行态都属于 `ContentBlock`
- 后端无需再区分“目录节点”和“内容块节点”两套路由

## 6.3 与现有接口的关系

### 可直接复用

- `backend/api/projects.py`
  - 现有项目完整 JSON 导出
- `backend/api/settings.py`
  - 现有 `FieldTemplate` 创建接口
- `backend/api/blocks.py`
  - 现有块树读取能力

### 需要新增

- Markdown 导出接口
- 范围 JSON 导出接口
- 范围转模板接口
- 项目 JSON 追加导入接口

### 不建议复用的路径

不建议让前端自己把整棵树读回来再本地组装模板或 Markdown，原因是：

- 容易和后端真实格式漂移
- 依赖修复逻辑会散落到前端
- 后续 Agent 若要复用，也无法统一走同一套底层能力

## 七、与自动拆分草稿体系的关系

当前项目已有：

- `ProjectStructureDraft`
- `project_structure_drafts` API
- `project_structure_compiler`

本次新增菜单能力不应直接导出草稿态对象，而应遵循以下规则：

### 项目级 JSON

- 继续走完整项目导出
- 其中自然包含 `project_structure_drafts`

### 项目级 / 组级 / 内容块级 Markdown

- 只导出正式内容树，不导出草稿态对象

### 保存为内容块模板

- 一律以正式 `ContentBlock` 树为输入
- 不直接以草稿对象为输入

### 项目级从 JSON 导入

- 第一版只导入 JSON 中的正式 `content_blocks`
- 不把源文件中的 `project_structure_drafts` 自动并入当前项目

原因：

- 模板是正式复用结构
- 草稿是应用前临时对象
- 两者语义必须分开

## 八、实施顺序建议

1. 先补后端统一导出与转模板服务。
2. 再补项目级与节点级 API。
3. 再抽象支持二级菜单的前端菜单组件。
4. 最后在 `progress-panel` 与 `block-tree` 接入三个新动作。
5. 再在项目级快捷入口接入 `从 JSON 导入`。

这样做的好处：

- 先把格式真相收敛到后端
- 前端只消费结果，不承担格式拼装
- 后续 Agent 若接入，也能直接复用后端能力

## 九、测试与验收

## 9.1 Markdown

- 项目级复制 / 下载能按树顺序输出完整 Markdown
- 组级复制 / 下载只输出该组子树
- 内容块级复制 / 下载能输出当前内容块标题与正文

## 9.2 JSON

- 项目级 JSON 仍符合当前完整项目导出格式
- 组级 JSON 能正确保留子树结构与内部依赖
- 内容块级 JSON 能正确导出单节点范围

## 9.3 保存为内容块模板

- 项目级保存后，生成的模板可被正常实例化到其他项目
- 组级保存后，模板根结构与原组一致
- 内容块级保存后，单节点模板可正常实例化
- 范围外依赖会给出明确提示，不会静默生成坏模板

## 9.4 项目级从 JSON 导入

- 能读取当前系统导出的项目 JSON 文件
- 导入后只在当前项目顶层末尾追加新节点
- 现有目录结构、顺序、内容都不受影响
- 导入子树内部依赖能正确重映射
- 源文件中范围外依赖会给出明确提示
- 不会把源项目本体覆盖到当前项目上

## 十、关键决策点

以下几点建议你审阅时重点拍板：

1. 项目级 JSON 是否确认继续使用“完整项目导出格式”，而不是另做“轻量内容树 JSON”。
2. 组级 / 内容块级 JSON 是否接受采用新的 `content_block_bundle` 范围导出格式。
3. “保存为内容块模板”遇到范围外依赖时，是否接受第一版只提示并丢弃外部依赖，而不是阻塞保存。
4. 项目级 `从 JSON 导入` 是否确认只支持“追加内容树”，不导入对话、草稿、评分器等外围对象。
5. 项目级三点菜单的位置，是放在项目标题右侧，还是和 `自动拆分内容` / `开始所有已就绪内容块` 归为同一操作区。

## 十一、推荐结论

本轮最稳妥、最符合当前项目结构的方案是：

- `复制为 Markdown 格式`
  - 三层统一走同一套 Markdown 序列化器
- `下载`
  - Markdown：沿用同一序列化结果
  - JSON：项目级复用现有完整导出，组级/内容块级新增范围导出格式
- `保存为内容块模板`
  - 三层统一把选中范围转换为 `FieldTemplate.root_nodes`
  - 项目级允许直接把整棵项目树沉淀为模板
- 项目级 `从 JSON 导入`
  - 放在左侧底部快捷入口区，不放进三点菜单
  - 第一版优先兼容当前系统完整项目导出 JSON
  - 导入行为固定为“追加到当前目录树后面，不影响现有目录结构”

这样可以做到：

- 不新造模板格式
- 不破坏现有项目导出格式
- 不把前端变成格式拼装中心
- 项目级、组级、内容块级三层语义统一但不强行同构
- “导入项目 JSON 到当前项目”被严格限定为内容树追加，避免误伤现有结构
